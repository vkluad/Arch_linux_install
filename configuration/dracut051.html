Глава 1. Определение

Электронный диск является временной файловой системой , используемой в процессе загрузки ядра Linux. initrd и initramfs относятся к немного разным схемам загрузки этой файловой системы в память. Оба обычно используются для подготовки перед монтированием настоящей корневой файловой системы.

Глава 2. Обоснование

Многие дистрибутивы Linux поставляют один общий образ ядра, предназначенный для загрузки как можно большего количества оборудования. Драйверы устройств для этого общего образа ядра включены в качестве загружаемых модулей, поскольку невозможно статически скомпилировать их все в одно ядро, не сделав его слишком большим для загрузки с компьютеров с ограниченной памятью или с носителей меньшей емкости, таких как дискеты.

Затем возникает проблема обнаружения и загрузки модулей, необходимых для монтирования корневой файловой системы во время загрузки (или, если на то пошло, определения того, где и что находится корневая файловая система).

Еще больше усложняет ситуацию то, что корневая файловая система может находиться на томе программного RAID, LVM, NFS (на бездисковых рабочих станциях) или в зашифрованном разделе. Все это требует специальной подготовки для установки.

Еще одна сложность - это поддержка спящего режима ядром, который переводит компьютер в режим ожидания на диске, выгружая образ всей системы в раздел подкачки или в обычный файл, а затем отключая питание. При следующей загрузке этот образ должен быть доступен, прежде чем его можно будет загрузить обратно в память.

Чтобы избежать необходимости жестко закодировать обработку такого количества особых случаев в ядре, используется начальная стадия загрузки с временной корневой файловой системой, которая теперь называется ранним пользовательским пространством. Эта корневая файловая система будет содержать помощников пользовательского пространства, которые будут выполнять обнаружение оборудования, загрузку модулей и обнаружение устройств, необходимых для монтирования настоящей корневой файловой системы.

Глава 3. Реализация

Образ этой исходной корневой файловой системы (вместе с образом ядра) должен храниться где-нибудь, доступном для загрузчика Linux или загрузочной прошивки компьютера. Это может быть:

  • Сама корневая файловая система
  • Загрузочный образ на оптическом диске
  • Небольшой раздел в формате ext2 / ext3 или FAT на локальном диске ( загрузочный раздел )
  • Сервер TFTP (в системах, которые могут загружаться через Ethernet)

Загрузчик загрузит ядро ​​и начальный образ корневой файловой системы в память, а затем запустит ядро, передав адрес памяти образа.

В зависимости от того, какие алгоритмы были статически скомпилированы в него, ядро ​​в настоящее время может распаковывать образы initrd / initramfs, сжатые с помощью gzip, bzip2 и LZMA.

Глава 4. Подготовка к установке

dracut может сгенерировать настроенный образ initramfs, который содержит только то, что необходимо для загрузки определенного компьютера, например, модули ядра ATA, SCSI и файловой системы (режим только для хоста).

dracut также может генерировать более общий образ initramfs (режим по умолчанию).

initramfs dracut запускается только с имени устройства корневой файловой системы (или его UUID) и должен обнаруживать все остальное во время загрузки. Для монтирования корневой файловой системы необходимо выполнить сложный каскад задач:

  • Все драйверы оборудования, от которых зависит процесс загрузки, должны быть загружены. Все модули ядра для обычных устройств хранения упаковываются в initramfs, а затем udev загружает модули, соответствующие обнаруженному компьютерному оборудованию.
  • В системах, которые отображают экран загрузки rd.splash, видеооборудование должно быть инициализировано, и помощник пользовательского пространства начал рисовать анимацию на дисплее синхронно с процессом загрузки.

  • Если корневая файловая система находится на NFS, dracut делает следующее:

    • Поднимите основной сетевой интерфейс.
    • Вызов DHCP-клиента, с помощью которого он может получить аренду DHCP.
    • Извлеките имя общего ресурса NFS и адрес сервера NFS из аренды.
    • Смонтируйте общий ресурс NFS.
  • Если корневая файловая система находится на программном RAID-устройстве, невозможно узнать, какие устройства охватывает том RAID; должны быть вызваны стандартные утилиты MD, чтобы просканировать все доступные блочные устройства с рейдовой сигнатурой и вывести требуемые в оперативный режим.
  • Если кажется, что корневая файловая система находится на логическом томе, необходимо вызвать утилиты LVM для поиска и активации группы томов, содержащей ее.

  • Если корневая файловая система находится на зашифрованном блочном устройстве:

    • Вызов вспомогательного сценария, предлагающего пользователю ввести парольную фразу и / или вставить аппаратный токен (например, смарт-карту или защитный ключ USB).
  • Создайте цель дешифрования с помощью устройства сопоставления.

dracut использует udev, агент горячего подключения, управляемый событиями, который вызывает вспомогательные программы при подключении оборудования, дисковых разделов и томов хранения, соответствующих определенным правилам. Это позволяет выполнять обнаружение параллельно и постепенно каскадировать в произвольные вложения LVM, RAID или шифрования для доступа к корневой файловой системе.

Когда корневая файловая система наконец станет видимой:

  • Все задачи обслуживания, которые невозможно запустить в смонтированной корневой файловой системе, выполнены.
  • Корневая файловая система монтируется только для чтения.
  • Все процессы, которые должны продолжать работать (например, помощник экрана rd.splash и его команда FIFO), переносятся во вновь смонтированную корневую файловую систему.

Окончательная корневая файловая система не может быть просто смонтирована поверх /, так как это сделало бы сценарии и инструменты исходной корневой файловой системы недоступными для любых задач окончательной очистки. В initramfs исходная корневая файловая система не может быть повернута. Вместо этого он просто очищается, а последняя корневая файловая система монтируется поверх.

Если в initramfs используется модуль systemd, порядок запускаемых служб выглядит как в главе 13, DRACUT.BOOTUP (7) .

Глава 5. Dracut при выключении

В системе, управляемой systemd, dracut initramfs также используется для процедуры завершения работы.

Во время выключения выполняются следующие шаги:

  • systemd переключается на shutdown.target
  • systemd запускает $ prefix / lib / systemd / system / shutdown.target.wants / dracut-shutdown.service
  • dracut-shutdown.service выполняет / usr / lib / dracut / dracut-initramfs-restore, который распаковывает initramfs в / run / initramfs
  • systemd завершает shutdown.target
  • systemd убивает все процессы
  • systemd пытается все размонтировать и монтирует оставшиеся только для чтения
  • systemd проверяет наличие исполняемого файла / run / initramfs / shutdown
  • если да, он делает pivot_root для / run / initramfs и выполняет ./shutdown. Затем старый корень монтируется в / oldroot. /usr/lib/dracut/modules.d/99shutdown/shutdown.sh - исполняемый файл завершения работы.
  • shutdown попытается размонтировать все монтирования / oldroot и вызовет различные хуки выключения из модулей dracut

Таким образом обеспечивается чистая разборка и демонтаж всех устройств.

ИМЯ

dracut - низкоуровневый инструмент для генерации образа initramfs / initrd

ОБЗОР

dracut [ ВАРИАНТ… ] [ <изображение> [ <версия ядра> ]]

ОПИСАНИЕ

Создайте initramfs <image> для ядра с версией <kernel version>. Если <версия ядра> опущено, то используется версия фактически работающего ядра. Если <изображение> опущено или пусто, то используется расположение по умолчанию / boot / initramfs- <версия ядра> .img.

dracut создает начальный образ, используемый ядром для предварительной загрузки модулей блочных устройств (таких как IDE, SCSI или RAID), которые необходимы для доступа к корневой файловой системе, монтирования корневой файловой системы и загрузки в реальную систему.

Во время загрузки ядро ​​распаковывает этот архив на RAM-диск, монтирует и использует его в качестве начальной корневой файловой системы. Все поиски корневого устройства происходят в этом раннем пользовательском пространстве.

Образы initramfs также называются «initrd».

Полный список параметров командной строки ядра см. В dracut.cmdline (7).

Если вы попали в аварийную оболочку при загрузке initramfs, создается файл /run/initramfs/rdsosreport.txt , который можно сохранить в разделе (для ручного монтирования) (обычно / boot) или на USB-накопителе. . Дополнительную отладочную информацию можно получить, добавив rd.debug в командную строку ядра. /run/initramfs/rdsosreport.txt содержит все журналы и выходные данные некоторых инструментов. Его следует прикреплять к любому отчету о проблемах с дракутом.

ИСПОЛЬЗОВАНИЕ

Самая простая команда для создания образа initramfs:

# дракута

В результате будет сгенерирован образ initramfs общего назначения со всеми возможными функциями, являющимися результатом комбинации установленных модулей dracut и системных инструментов. Образ находится в / boot / initramfs- <kernel version>.img и содержит модули ядра текущего активного ядра с указанием версии <kernel version>.

Если образ initramfs уже существует, dracut отобразит сообщение об ошибке, и чтобы перезаписать существующий образ, вы должны использовать параметр --force.

# dracut --force

Если вы хотите указать другое имя файла для результирующего изображения, вы должны ввести команду вроде:

# dracut foobar.img

Чтобы сгенерировать образ для конкретной версии ядра, используйте следующую команду:

# dracut foobar.img 2.6.40-1.rc5.f20

Ярлык для создания образа в местоположении по умолчанию для конкретной версии ядра:

# dracut --kver 2.6.40-1.rc5.f20

Если вы хотите создать более легкие и маленькие образы initramfs, вы можете указать параметр --hostonly или -H. При использовании этой опции результирующий образ будет содержать только те модули dracut, модули ядра и файловые системы, которые необходимы для загрузки этой конкретной машины. Это имеет недостаток, заключающийся в том, что вы не можете поместить диск на другой контроллер или машину и что вы не можете переключиться на другую корневую файловую систему, не воссоздавая образ initramfs. Использование опции --hostonly предназначено только для экспертов, и вам придется хранить осколки. По крайней мере, сохраните копию образа общего назначения (и соответствующего ядра) в качестве запасного варианта для спасения вашей системы.

Проверка содержимого

Чтобы увидеть содержимое изображения, созданного dracut, вы можете использовать инструмент lsinitrd.

# lsinitrd | меньше

Для отображения содержимого файла в initramfs также используйте инструмент lsinitrd:

# lsinitrd -f /etc/ld.so.conf 
include ld.so.conf.d / *. conf

Добавление модулей dracut

Некоторые модули dracut по умолчанию отключены, и их нужно активировать вручную. Вы можете сделать это, добавив модули dracut в файл конфигурации /etc/dracut.conf или /etc/dracut.conf.d/myconf.conf . См. Dracut.conf (5). Вы также можете добавить модули dracut в командную строку с помощью параметра -a или --add:

# dracut --add bootchart initramfs-bootchart.img

Чтобы увидеть список доступных модулей dracut, используйте параметр --list-modules:

# dracut --list-modules

Отсутствие модулей dracut

Иногда вы не хотите, чтобы модуль dracut был включен по причинам скорости, размера или функциональности. Для этого либо укажите переменную omit_dracutmodules в файле конфигурации dracut.conf или /etc/dracut.conf.d/myconf.conf (см. Dracut.conf (5)), либо используйте параметр -o или --omit в командная строка:

# dracut -o "multipath lvm" no-multipath-lvm.img

Добавление модулей ядра

Если вам нужен специальный модуль ядра в initramfs, который не выбирается автоматически dracut, вы можете использовать параметр --add-drivers в командной строке или переменную драйверов в файле /etc/dracut.conf или / etc Файл конфигурации /dracut.conf.d/myconf.conf (см. dracut.conf (5)):

# dracut --add-drivers mymod initramfs-with-mymod.img

Параметры загрузки

Initramfs, сгенерированный без режима «hostonly», не содержит файлов конфигурации системы (за исключением некоторых особых исключений), поэтому конфигурация должна выполняться из командной строки ядра. Благодаря такой гибкости вы можете легко загрузиться с измененного корневого раздела без необходимости перекомпилировать образ initramfs. Таким образом, вы можете полностью изменить свой корневой раздел (переместить его в рейд MD с шифрованием и LVM поверх), если вы укажете правильную файловую систему LABEL или UUID в командной строке ядра для вашего корневого устройства, dracut найдет его и загрузиться с него.

Командная строка ядра также может быть предоставлена ​​сервером DHCP с параметром root-path. См. Раздел «Сетевая загрузка» .

Полный справочник по всем параметрам командной строки ядра см. В dracut.cmdline (5).

Чтобы быстро начать работу с подходящей командной строкой ядра в вашей системе, используйте параметр --print-cmdline :

# dracut --print-cmdline 
 root = UUID = 8b8b6f91-95c7-4da2-831b-171e12179081 rootflags = rw, relatime, discard, data = order rootfstype = ext4

Указание корневого устройства

Это единственный вариант, который действительно необходим dracut для загрузки с корневого раздела. Поскольку ваш корневой раздел может находиться в различных средах, для параметра root = существует множество форматов. Самый простой - root = <path to device node>:

корень = / dev / sda2

Поскольку имена узлов устройства могут меняться в зависимости от порядка дисков, рекомендуется использовать идентификатор файловой системы (UUID) или метку файловой системы (LABEL) для указания корневого раздела:

корень = UUID = 19e9dda3-5a38-484d-a9b0-fa6b067d0331

или же

root = LABEL = myrootpartitionlabel

Чтобы увидеть все UUID или LABEL в вашей системе, выполните:

# ls -l / dev / disk / by-uuid

или же

# ls -l / dev / disk / по метке

Если ваш корневой раздел находится в сети, см. Раздел «Сетевая загрузка» .

Настройки клавиатуры

Если вам нужно ввести пароли для зашифрованных дисковых томов, вы можете настроить раскладку клавиатуры и указать шрифт отображения.

Типичная немецкая командная строка ядра будет содержать:

rd.vconsole.font = eurlatgr rd.vconsole.keymap = de-latin1-nodeadkeys rd.locale.LANG = de_DE.UTF-8

Установка этих параметров может переопределить параметр, хранящийся в вашей системе, если вы используете современную систему инициализации, такую ​​как systemd.

Внесение модулей ядра в черный список

Иногда требуется предотвратить автоматическую загрузку модуля ядра для определенного модуля ядра. Чтобы сделать это, просто добавьте rd.blacklist = <kernel module name>с <kernel module name>не содержащей .ko суффикс, в командной строке ядра. Например:

rd.driver.blacklist = mptsas rd.driver.blacklist = nouveau

Опцию можно указать несколько раз в командной строке ядра.

Ускорение процесса загрузки

Если вы хотите ускорить процесс загрузки, вы можете указать как можно больше информации для dracut в команде ядра. Например, вы можете сообщить dracut, что ваш корневой раздел не находится на томе LVM или не находится в разделе raid, или что он находится внутри определенного зашифрованного тома LUKS. По умолчанию dracut ищет везде. Типичная командная строка ядра dracut для простого первичного или логического раздела будет содержать:

rd.luks = 0 rd.lvm = 0 rd.md = 0 rd.dm = 0

Это отключает каждую автоматическую сборку LVM, MD-рейдов, DM-рейдов и криптографических LUKS.

Конечно, вы также можете опустить модули dracut в процессе создания initramfs, но тогда вы потеряете возможность включать их по запросу.

Внедрение пользовательских файлов

Чтобы добавить свои собственные файлы в образ initramfs, у вас есть несколько возможностей.

Параметр --include позволяет указать исходный и целевой путь. Например

# dracut --include cmdline-preset /etc/cmdline.d/mycmdline.conf initramfs-cmdline-pre.img

создаст образ initramfs, где файл cmdline-preset будет скопирован внутри initramfs в /etc/cmdline.d/mycmdline.conf . --include можно указать только один раз.

# mkdir -p rd.live.overlay / etc / cmdline.d 
# mkdir -p rd.live.overlay / etc / conf.d 
# echo "ip = dhcp" >> rd.live.overlay / etc / cmdline.d /mycmdline.conf 
# экспорт эха FOO = testtest >> rd.live.overlay / etc / conf.d / testvar.conf 
# экспорт эхо BAR = testtest >> rd.live.overlay / etc / conf.d / testvar.conf 
# tree rd.live.overlay / 
rd.live.overlay / 
`- etc 
    | - cmdline.d 
    | `- mycmdline.conf` 
    - conf.d 
        `- testvar.conf 

# dracut --include rd.live.overlay / initramfs-rd.live.overlay.img

Это поместит содержимое каталога rd.live.overlay в корень образа initramfs.

Параметр --install позволяет указать несколько файлов, которые будут установлены в образе initramfs в одном месте, так как они присутствовали во время создания initramfs.

# dracut --install 'strace fsck.ext3 ssh' initramfs-dbg.img

Это создаст initramfs с исполняемыми файлами strace, fsck.ext3 и ssh вместе с библиотеками, необходимыми для их запуска. Параметр --install можно указывать несколько раз.

Сетевая загрузка

Если ваш корневой раздел находится на сетевом диске, у вас должны быть установлены сетевые модули dracut, чтобы создать сетевой образ initramfs.

Если вы укажете ip = dhcp в командной строке ядра, то dracut запросит у dhcp-сервера IP-адрес машины. Сервер DHCP также может служить дополнительным корневым путем, который устанавливает корневое устройство для dracut. С помощью этого механизма у вас есть статическая конфигурация на вашем клиентском компьютере и централизованная конфигурация загрузки на сервере TFTP / DHCP. Если вы не можете передать командную строку ядра, вы можете ввести /etc/cmdline.d/mycmdline.conf с помощью метода, описанного в разделе «Внедрение пользовательских файлов» .

Уменьшение размера изображения

Чтобы уменьшить размер initramfs, вы должны создать его, опуская все модули dracut, которые, как вы знаете, вам не нужно загружать.

Вы также можете указать точные модули dracut и ядра для создания очень маленького образа initramfs.

Например, для образа NFS вы должны сделать:

# dracut -m "сетевая база nfs" initramfs-nfs-only.img

Затем вы загрузитесь с этого образа на своей целевой машине и уменьшите размер еще раз, создав его на целевой машине с параметром --host-only:

# dracut -m "сетевая база nfs" --host-only initramfs-nfs-host-only.img

Это значительно уменьшит размер образа initramfs.

Исправление проблем

Если процесс загрузки не удался, у вас есть несколько вариантов отладки ситуации. Здесь описаны некоторые из основных операций. Для получения дополнительной информации посетите также: https://www.kernel.org/pub/linux/utils/boot/dracut/dracut.html

Определение вашей проблемной области

  1. Удалите 'rhgb ' и 'quiet ' из командной строки ядра
  2. Добавьте 'rd.shell ' в командную строку ядра. Это представит оболочку, если dracut не сможет найти ваше корневое устройство.
  3. Добавьте 'rd.shell rd.debug log_buf_len = 1M ' в командную строку ядра, чтобы команды оболочки dracut выводились по мере их выполнения
  4. Создается файл /run/initramfs/rdsosreport.txt, который содержит все журналы и выходные данные всех важных инструментов, которые упомянуты позже.

Если вы хотите сохранить этот вывод, просто смонтируйте / загрузитесь вручную или вставьте USB-накопитель и смонтируйте его. Затем вы можете сохранить результат для последующей проверки.

Доступ к корневому тому из оболочки дракута

Из оболочки отладки dracut вы можете вручную выполнить задачу поиска и подготовки корневого тома к загрузке. Необходимые шаги будут зависеть от того, как настроен ваш корневой том. Общие сценарии включают:

  • Блочное устройство (например, / dev / sda7 )
  • Логический том LVM (например, / dev / VolGroup00 / LogVol00 )
  • Зашифрованное устройство (например, / dev / mapper / luks-4d5972ea-901c-4584-bd75-1da802417d83 )
  • Подключенное к сети устройство (например, netroot = iscsi: @ 192.168.0.4 :: 3260 :: iqn.2009-02.org.example: for.all )

Точный метод поиска и подготовки будет отличаться. Однако для продолжения успешной загрузки цель состоит в том, чтобы найти корневой том и создать символическую ссылку / dev / root, указывающую на файловую систему. Например, в следующем примере демонстрируется доступ и загрузка корневого тома, который представляет собой зашифрованный логический том LVM.

  1. Осмотрите свои разделы с помощью parted 

    # parted / dev / sda -sp 
Модель: ATA HTS541060G9AT00 (scsi) 
Диск / dev / sda: 60,0 ГБ 
Размер сектора (логический / физический): 512B / 512B

Номер Начало Конец Размер Тип Флаги файловой системы 
1 32,3 КБ 10,8 107 МБ основная загрузка ext4 
2 10,8 ГБ 55,6 ГБ 44,7 ГБ логический lvm

  2. Вы помните, что ваш корневой том был логическим томом LVM. Сканируйте и активируйте любые логические тома. 

    # lvm vgscan 
# lvm vgchange -ay

  3. Теперь вы должны увидеть все логические тома, используя команду blkid: 

    # blkid

/ dev / sda2: UUID = "Ek4dQw-cOtq-5MJu-OGRF-xz5k-O2l8-wdDE0I =" LVM2_member " 
/ dev / mapper / linux-root: UUID =" def0269e-424b-4752-acf3-1077bf96ad2c "TYPE =" crypto_LUKS " 
/ dev / mapper / linux-home: UUID =" c69127c1-f153-4ea2-b589e-4c "TYPE =" ext3 " 
/ dev / mapper / linux-swap: UUID =" 47b4d329-975c-4c08-b218-f9c9bf3635f1 "TYPE =" swap "

  4. Из вышеприведенного вывода вы помните, что ваш корневой том существует на зашифрованном блочном устройстве. Следуя инструкциям по шифрованию диска из Руководства по установке, вы разблокируете зашифрованный корневой том. 

    # UUID = $ (cryptsetup luksUUID / dev / mapper / linux-root) 
# cryptsetup luksOpen / dev / mapper / linux-root luks- $ UUID 
Введите кодовую фразу для / dev / mapper / linux-root: 
ключевой слот 0 разблокирован.

    # ln -s / dev / mapper / luks- $ UUID / dev / root

    # выход

dracut.conf - файл (ы) конфигурации для dracut

ОБЗОР

/etc/dracut.conf /etc/dracut.conf.d/*.conf /usr/lib/dracut/dracut.conf.d/*.conf

Описание

dracut.conf загружается на этапе инициализации dracut. Параметр командной строки имеет приоритет над любыми значениями, установленными здесь.

Файлы * .conf читаются из /usr/lib/dracut/dracut.conf.d и /etc/dracut.conf.d. Файлы с тем же именем в /etc/dracut.conf.d заменят файлы в /usr/lib/dracut/dracut.conf.d. Затем файлы читаются в алфавитно-цифровом порядке и переопределяют параметры, установленные в /etc/dracut.conf . Каждая строка определяет атрибут и значение. # Указывает на начало комментария; следующие символы до конца строки не интерпретируются.

Параметры командной строки dracut имеют приоритет над любыми значениями, установленными здесь.

Файлы конфигурации должны иметь расширение .conf; другие расширения игнорируются.

add_dracutmodules + = "  <модули dracut>  "
Добавьте разделенный пробелами список модулей dracut для вызова при сборке initramfs. Модули находятся в /usr/lib/dracut/modules.d .
dracutmodules + = "  <модули dracut>  "
Укажите разделенный пробелами список модулей dracut для вызова при сборке initramfs. Модули находятся в /usr/lib/dracut/modules.d . Эта опция заставляет dracut включать только указанные модули dracut. В большинстве случаев вы хотите использовать параметр «add_dracutmodules».
omit_dracutmodules + = "  <модули dracut>  "
Опустите список модулей dracut, разделенных пробелами, для вызова при сборке initramfs. Модули находятся в /usr/lib/dracut/modules.d .
драйверы + = "  <модули ядра>  "
Задайте список модулей ядра, разделенных пробелами, для исключительного включения в initramfs. Модули ядра следует указывать без суффикса «.ko».
add_drivers + = "  <модули ядра>  "
Задайте список модулей ядра, разделенных пробелами, для добавления в initramfs. Модули ядра следует указывать без суффикса «.ko».
force_drivers + = "  <список модулей ядра>  "
См. Add_drivers выше. Но в этом случае гарантируется, что драйверы будут загружены раньше через modprobe.
omit_drivers + = "  <модули ядра>  "
Укажите разделенный пробелами список модулей ядра, которые не следует добавлять в initramfs. Модули ядра следует указывать без суффикса «.ko».
filesystems + = "  <имена файловых систем>  "
Задайте список модулей файловой системы ядра, разделенных пробелами, для исключительного включения в общий initramfs.
drivers_dir = " <каталог модулей ядра> "
Укажите каталог, в котором искать модули ядра
fw_dir + = ": < каталог > [: < каталог >  …]"
Укажите дополнительные каталоги, где искать прошивки, через:
install_items + = "  <файл> [ <файл>  …]"
Укажите дополнительные файлы для включения в initramfs, разделив их пробелами.
install_optional_items + = "  <файл> [ <файл>  …]"
Укажите дополнительные файлы для включения в initramfs, разделенные пробелами, если они существуют.
compress = " {cat | bzip2 | lzma | xz | gzip | lzo | lz4 | zstd | <компрессор [аргументы…]>} "
Сжать сгенерированные файлы initramfs с помощью переданной программы сжатия. Если вы передадите ему просто имя программы сжатия, она вызовет эту программу с известными рабочими аргументами. Если вы передадите аргументы, он будет вызываться именно с этими аргументами. В зависимости от того, что вы передадите, это может привести к тому, что ядро ​​не сможет распаковать initramfs. Чтобы отключить сжатие, используйте «cat».
do_strip = " {да | нет} "
Убрать двоичные файлы в initramfs (по умолчанию = да)
hostonly = " {да | нет} "
Режим только для хоста: установите только то, что необходимо для загрузки локального хоста вместо общего хоста, и создайте конфигурацию для конкретного хоста.
hostonly_cmdline = " {да | нет} "
Если установлено «да», сохранить необходимые аргументы командной строки ядра в initramfs.
persistent_policy = " <политика> "
Используйте <policy> для адресации дисков и разделов. <policy> может быть любым именем каталога, найденным в / dev / disk. Например, "by-uuid", "by-label"
tmpdir = " <временный каталог> "
Укажите временный каталог для использования.

use_fstab = " {да | нет} "
Используйте / etc / fstab вместо / proc / self / mountinfo .

Добавьте записи <filename> в файл initramfs / etc / fstab.
add_device + = "  <устройство>  "
Вызовите <устройство> в initramfs, <устройство> должно быть именем устройства. Это может быть полезно в режиме hostonly для возобновления поддержки, когда ваш своп находится на LVM зашифрованном разделе.

Включить локальный /etc/mdadm.conf (по умолчанию = да)

Добавьте список инструментов fsck, разделенных пробелами. Если ничего не указано, значение по умолчанию: «umount mount / sbin / fsck * xfs_db xfs_check xfs_repair e2fsck jfs_fsck reiserfsck btrfsck». Установка возможна (несуществующие инструменты игнорируются).
nofscks = " {да | нет} "
Если указано, запретите установку любых инструментов fsck.
ro_mnt = " {да | нет} "
По умолчанию монтируйте / и / usr только для чтения.
kernel_cmdline = " параметры "
Укажите параметры командной строки ядра по умолчанию
kernel_only = " {да | нет} "
Устанавливайте только драйверы ядра и файлы прошивки. (по умолчанию = нет)
no_kernel = " {да | нет} "
Не устанавливать драйверы ядра и файлы прошивки (по умолчанию = нет)
acpi_override = " {да | нет} "
[ВНИМАНИЕ] ИСПОЛЬЗУЙТЕ ЭТО ТОЛЬКО, ЕСЛИ ВЫ ЗНАЕТЕ, ЧТО ВЫ ДЕЛАЕТЕ! Переопределение BIOS предоставило таблицы ACPI. Для получения дополнительной документации прочтите Documentation / acpi / initrd_table_override.txt в исходных кодах ядра. Найдите файлы таблиц ACPI (должны иметь суффикс .aml) в каталоге acpi_table_dir = (см. Ниже) и добавьте их в отдельный несжатый архив cpio. Этот архив cpio приклеивается (объединенный, несжатый архив должен быть первым) к сжатому архиву cpio. Первый несжатый архив cpio предназначен для данных, к которым ядро ​​должно иметь доступ на очень раннем этапе (и пока не может использовать алгоритмы распаковки), например, микрокод или таблицы ACPI (по умолчанию = нет).
acpi_table_dir = " <dir> "
Каталог для поиска таблиц ACPI, если acpi_override = имеет значение yes.
Early_microcode = "{да | нет}"
Объединить ранний микрокод с ramdisk (по умолчанию = да)
stdloglvl = " {0-6} "
Установите для ведения журнала стандартный уровень ошибок.
sysloglvl = " {0-6} "
Установите ведение журнала на уровень системного журнала.
fileloglvl = " {0-6} "
Установите ведение журнала на уровне файла.
logfile = " <файл> "
Путь к файлу журнала.
show_modules = " {да | нет} "
Выведите имя включенных модулей в стандартный вывод во время сборки.
i18n_vars = " <отображение переменных> "
Отображение переменных, специфичных для распределения. Подробное описание см. В dracut / modules.d / 10i18n / README.
i18n_default_font = " <fontname> "
Шрифт <fontname> для установки, если не указано иное. По умолчанию - «eurlatgr».
i18n_install_all = " {да | нет} "
Устанавливайте все независимо от общего режима или режима только для хоста.
reproducible = " {да | нет} "
Создавайте воспроизводимые изображения.
loginstall = " < КАТАЛОГ > "
Зарегистрируйте все файлы, установленные с хоста, в <DIR> .
uefi_stub = " <ФАЙЛ> "
Задает загрузчик-заглушку UEFI, который загружает подключенное ядро, initramfs и командную строку ядра и загружает ядро. По умолчанию это /lib/systemd/boot/efi/linux<EFI-MACHINE-TYPE-NAME>.efi.stub или /usr/lib/gummiboot/linux<EFI-MACHINE-TYPE-NAME>.efi.stub
uefi_splash_image = " <ФАЙЛ> "
Задает заставку загрузчика заглушек UEFI. Требуется формат растрового изображения ( .bmp ).
uefi_secureboot_cert = " <FILE> ", uefi_secureboot_key = " <FILE> "
Задает сертификат и соответствующий ключ, которые используются для подписи созданного исполняемого файла UEFI. Требуется указать сертификат и ключ, а также установить подпись .
kernel_image = " <ФАЙЛ> "
Задает образ ядра, который нужно включить в исполняемый файл UEFI. По умолчанию это / lib / modules / <KERNEL-VERSION> / vmlinuz или / boot / vmlinuz- <KERNEL-VERSION>.

Файлы

/etc/dracut.conf
Старый файл конфигурации. Лучше использовать свой собственный файл в /etc/dracut.conf.d/ .
/etc/dracut.conf.d/
Любой файл /etc/dracut.conf.d/*.conf может переопределять значения в /etc/dracut.conf . Файлы конфигурации читаются в алфавитно-цифровом порядке.

dracut.cmdline - параметры командной строки ядра dracut

ОПИСАНИЕ

Как всегда, корневое устройство, используемое ядром, указывается в файле конфигурации загрузки в командной строке ядра.

Традиционная спецификация устройства в стиле root = / dev / sda1 разрешена, но не рекомендуется. Корневое устройство лучше идентифицировать по LABEL или UUID. Если используется метка, например root = LABEL = <label_of_root>, initramfs будет искать на всех доступных устройствах файловую систему с соответствующей меткой и монтировать это устройство как корневую файловую систему. root = UUID = <uuidnumber> смонтирует раздел с этим UUID в качестве корневой файловой системы.

Далее описаны все параметры командной строки ядра, которые обрабатываются dracut.

Параметры «rd. *», указанные без «=», являются логическими параметрами. Их можно включить / выключить, установив для них значение {0 | 1}. Если присвоение с "=" отсутствует, подразумевается "= 1". Например, rd.info можно отключить с помощью rd.info = 0 или включить с помощью rd.info = 1 или rd.info . Последнее значение в командной строке ядра - это значение, которое учитывается.

Стандарт

init = <путь к настоящему init>
укажите путь к программе инициализации, которая будет запущена после завершения initramfs
root = <путь к блочному устройству>

укажите блочное устройство, которое будет использоваться в качестве корневой файловой системы.

Пример.  

root = / dev / sda1 
root = / dev / disk / by-path / pci-0000: 00: 1f.1-scsi-0: 0: 1: 0-part1 
root = / dev / disk / by-label / Root 
root = LABEL = Корень 
root = / dev / disk / by-uuid / 3f5ad593-4546-4a94-a374-bcfb68aa11f7 
root = UUID = 3f5ad593-4546-4a94-a374-bcfb68aa11f7 
root = PARTUUID = 3f5-4ada94-45 bcfb68aa11f7

rootfstype = <тип файловой системы>

"авто", если не указано иное.

Пример.  

rootfstype = ext3

rootflags = <параметры монтирования>
укажите дополнительные параметры монтирования для корневой файловой системы. Если не задан, / etc / fstab настоящего корня будет проанализирован на предмет специальных параметров монтирования и соответственно смонтирован.
ро
принудительно монтировать / и / usr (если это отдельное устройство) только для чтения. Если ни один из ro и rw не присутствует, оба монтируются в соответствии с / etc / fstab .
rw
принудительно монтировать / и / usr (если это отдельное устройство) чтение-запись. См. Также вариант ro.
rootfallback = <путь к блочному устройству>
укажите блочное устройство, которое будет использоваться в качестве корневой файловой системы, если нормальный корень не может быть найден. Это может быть только простое блочное устройство с простой файловой системой, для которой драйвер файловой системы либо скомпилирован, либо добавлен вручную в initramfs. Этот параметр можно указывать несколько раз.
rd.auto rd.auto = 1
включить автосборку специальных устройств, таких как cryptoLUKS, dmraid, mdraid или lvm. По умолчанию для версии dracut> = 024 отключено.
rd.hostonly = 0
удаляет все скомпилированные в конфигурации хост-системы, на которой был построен образ initramfs. Это помогает при загрузке, если структура диска изменилась, особенно в сочетании с rd.auto или другими параметрами, определяющими структуру.
rd.cmdline = спросить
запрашивает у пользователя дополнительные параметры командной строки ядра
rd.fstab = 0
не соблюдайте особые параметры монтирования для корневой файловой системы, найденные в / etc / fstab настоящего корня.
возобновить = <путь для возобновления раздела>

возобновить с раздела подкачки

Пример.  

возобновить = / dev / disk / by-path / pci-0000: 00: 1f.1-scsi-0: 0: 1: 0-part1 
resume = / dev / disk / by-uuid / 3f5ad593-4546-4a94-a374 -bcfb68aa11f7 
резюме = UUID = 3f5ad593-4546-4a94-a374-bcfb68aa11f7

rd.skipfsck
пропустите fsck для rootfs и / usr . Если вы монтируете / usr только для чтения, а система инициализации выполняет fsck перед повторным подключением, вы можете использовать эту опцию, чтобы избежать дублирования.

iso-scan / имя файла

Смонтируйте все монтируемые устройства и найдите ISO, указанный аргументом. Когда ISO найден, настройте его как устройство петли. Устройство, содержащее этот ISO-образ, останется смонтированным в / run / initramfs / isoscandev. Использование iso-scan / filename с iso Fedora / Red Hat / CentOS Live должно работать только путем копирования исходных параметров командной строки ядра.

Пример.  

menuentry 'Live Fedora 20' --class fedora --class gnu-linux --class gnu --class os { 
    set изолабель = Fedora-Live-LXDE-x86_64-20-1 
    set isofile = "/ boot / iso / Fedora- Live-LXDE-x86_64-20-1.iso " 
    loopback loop $ isofile 
    linux (loop) / isolinux / vmlinuz0 boot = isolinux iso-scan / filename = $ isofile root = live: LABEL = $ isolabel ro rd.live.image quiet rhgb 
    initrd (цикл) /isolinux/initrd0.img 
}

Разное

rd.emergency = [перезагрузка | poweroff | остановка]
укажите, какое действие выполнять в случае критического сбоя. Также необходимо указать rd.shell = 0.
rd.driver.blacklist = <drivername> [, <drivername> ,…]
не загружайте модуль ядра <drivername>. Этот параметр можно указывать несколько раз.
rd.driver.pre = <drivername> [, <drivername> ,…]
принудительно загрузить модуль ядра <drivername>. Этот параметр можно указывать несколько раз.
rd.driver.post = <drivername> [, <drivername> ,…]
принудительно загрузить модуль ядра <drivername> после загрузки всех модулей автоматической загрузки. Этот параметр можно указывать несколько раз.
rd.retry = <секунды>
укажите, как долго dracut должен повторять очередь инициализации для настройки устройств. По умолчанию 30 секунд. По прошествии 2/3 времени начнутся рейды с деградацией. Если у вас есть оборудование, для которого требуется очень много времени, чтобы объявить о своих дисках, вы можете увеличить это значение.
rd.timeout = <секунды>
указать, как долго dracut должен ждать появления устройств. По умолчанию 0 , что означает навсегда . Обратите внимание, что этот тайм-аут должен быть больше, чем rd.retry, чтобы обеспечить правильную настройку.
rd.noverifyssl
принимать самозаверяющие сертификаты для загрузок ssl.
rd.ctty = <оконечное устройство>
укажите управляющий терминал для консоли. Это полезно, если у вас есть несколько аргументов console =.

Отлаживать

Если вы попали в аварийную оболочку, создается файл /run/initramfs/rdsosreport.txt , который можно сохранить в раздел (для ручного монтирования) (обычно / boot) или на USB-накопитель. Дополнительную отладочную информацию можно получить, добавив rd.debug в командную строку ядра. /run/initramfs/rdsosreport.txt содержит все журналы и выходные данные некоторых инструментов. Его следует прикреплять к любому отчету о проблемах с дракутом.

rd.info
печатать информационный вывод, хотя установлен "тихий"
rd.shell
разрешить переход в оболочку, если монтирование root не удалось
rd.debug
установить -x для оболочки дракута. Если systemd активен в initramfs, весь вывод записывается в журнал systemd, который вы можете проверить с помощью "journalctl -ab". Если systemd не активен, журналы записываются в dmesg и /run/initramfs/init.log . Если установлено «тихо», он также записывается в консоль.
rd.memdebug = [0-5]

Распечатайте информацию об использовании памяти в различных точках, установите уровень детализации от 0 до 5.

rd.luks.key = <keypath> [: <keydev> [: <luksdev>]]

<keypath> - это путь к ключевому файлу относительно корня файловой системы на каком-либо устройстве. Это ОБЯЗАТЕЛЬНО. Когда <keypath> заканчивается на .gpg , считается, что ключ зашифрован симметрично GPG. Вам будет предложено ввести пароль GPG при загрузке. Поддержка GPG поставляется смодулем crypt-gpg , который необходимо добавить явно.

<keydev> идентифицирует устройство, на котором находится файл ключа. Это может быть имя ядра устройства (должно начинаться с «/ dev /»), UUID (с префиксом «UUID =») или метка (префикс с «LABEL =»). Вам не нужно указывать полный UUID. Достаточно только его начала, даже если оно неоднозначно. Все подходящие устройства будут проверены. Этот параметр рекомендуется, но не обязателен. Если его нет, будут проверены все блочные устройства, что может значительно увеличить время загрузки.

Если задано <luksdev> , указанный ключ будет использоваться только для указанного устройства LUKS. Возможные значения такие же, как для <keydev> . Если у вас нет нескольких устройств LUKS, вам не нужно указывать этот параметр. Самый простой способ использования:

Пример.  

rd.luks.key = / foo / bar.key

Как видите, в таком случае можно пропустить двоеточие.

Примечание

Ваш LUKS-раздел должен соответствовать вашему ключевому файлу.

dracut предоставляет ключи для cryptsetup с помощью -d (более старый псевдоним для --key-file ). Это использует все двоичное содержимое ключевого файла как часть секрета. Если вы передадите пароль в cryptsetup без -d или --key-file , он будет рассматриваться как вводимый пользователем текст, и будут использоваться только символы перед первой новой строкой . Поэтому, когда вы создаете зашифрованный раздел для монтирования dracut и передаете ключ в cryptsetup luksFormat , вы должны использовать -d - .

Вот пример ключа, зашифрованного с помощью GPG (предупреждение: --batch-mode перезапишет устройство без запроса подтверждения):

gpg --quiet --decrypt rootkey.gpg | \ 
cryptsetup --batch-mode --key-file - \ 
           luksFormat / dev / sda47

Если вы используете незашифрованные файлы ключей, просто используйте путь к файлу ключей вместо стандартного ввода. Для случайного ключа с 256 битами энтропии вы можете использовать:

голова -32c / dev / urandom> rootkey.key 
cryptsetup --batch-mode --key-file rootkey.key \ 
           luksFormat / dev / sda47

Вы также можете использовать обычные ключевые файлы в зашифрованном keydev .

По сравнению с использованием зашифрованных файлов ключей GPG на незашифрованном устройстве это дает следующие преимущества:

  • вы можете разблокировать свой диск (диски), используя несколько парольных фраз
  • лучшая безопасность за счет отсутствия потери механизма растяжения ключа

Чтобы использовать зашифрованный keydev, вы должны убедиться, что он становится доступным с помощью ключевого слова keysource, например, rd.luks.uuid=keysource:aaaa aaaa - это uuid зашифрованного keydev .

Пример:

Предположим, у вас есть три диска A , B и C с uuids aaaa , bbbb и cccc . Вы хотите разблокировать A и B с помощью ключевого файла keyfile . Разблокированные тома - это A ' , B' и C ' с uuid AAAA , BBBB и CCCC . keyfile сохраняется на C ' как / keyfile .

Один ключевой слот luks для каждого A , B и C настраивается с парольной фразой. Другой слот ключей luks каждого A и B настраивается с помощью ключевого файла .

Для загрузки этой конфигурации вы можете использовать:

rd.luks.uuid = aaaa 
rd.luks.uuid = bbbb 
rd.luks.uuid = keysource: cccc 
rd.luks.key = / keyfile: UUID = CCCC

Dracut запрашивает кодовую фразу для C и использует файл_ключа для разблокировки A и B . Если получение ключевой фразы для C не удается, возвращается к просить о фразах для A и B .

Если вы хотите, чтобы C ' оставался разблокированным, укажите для него имя luks, например rd.luks.name=cccc=mykeys, в противном случае он будет закрыт, когда больше не нужен.

Загрузка живых образов

Dracut предлагает несколько вариантов для изображений с прямой загрузкой:

SquashFS с образом файловой системы только для чтения

Система загрузится с файловой системой, доступной только для чтения, из SquashFS и применит доступный для записи моментальный снимок Device-mapper или оверлейное монтирование OverlayFS для базовой файловой системы только для чтения. Этот метод обеспечивает относительно быструю загрузку и меньшее использование оперативной памяти. Пользователи должны быть осторожны, чтобы не записывать слишком много блоков в том моментального снимка. После того, как блоки наложения моментальных снимков исчерпаны, корневая файловая система становится доступной только для чтения и может вызвать сбои приложений. Файл оверлея моментального снимка помечен как Переполнение , и требуется сложное восстановление, чтобы восстановить и увеличить оверлей в автономном режиме. Непостоянные наложения - это разреженные файлы в ОЗУ, которые занимают пространство содержимого только по мере выделения необходимых блоков. По умолчанию они имеют видимый размер 32 ГиБ в ОЗУ. Размер можно отрегулировать с помощью rd.live.overlay.size = параметр командной строки ядра.

Традиционно ожидается, что структура файловой системы будет следующей:

squashfs.img | SquashFS с LiveCD .iso 
   ! ( 
   Монтировать ) / LiveOS 
       | - rootfs.img | Образ файловой системы для монтирования только для чтения 
            ! (Монтирование) 
            / bin | Живая файловая система 
            / загрузка | 
            / dev | 
            ... |

Для оверлеев монтирования OverlayFS структура файловой системы также может быть прямым сжатием корневой файловой системы:

squashfs.img | SquashFS с LiveCD .iso 
   ! (Монтировать) 
   / bin | Живая файловая система 
   / загрузка | 
   / dev | 
   ... |

По умолчанию Dracut использует один из оверлейных методов живой загрузки. Никаких дополнительных параметров командной строки не требуется, кроме root = live: <URL>, чтобы указать расположение вашей сжатой файловой системы.

  • Сжатый образ SquashFS можно скопировать во время загрузки в ОЗУ /run/initramfs/squashed.imgс помощью параметра rd.live.ram = 1 .
  • Устройство с постоянным оверлеем можно загрузить в режиме « только чтение» с помощью параметра rd.live.overlay.readonly в командной строке ядра. Это либо приведет к наложению временного наложения с возможностью записи поверх снимка корневой файловой системы, доступного только для чтения, либо при монтировании OverlayFS будет использоваться дополнительный нижний уровень с корневой файловой системой.
Несжатый живой образ файловой системы

Когда живая система была установлена ​​с параметром --skipcompress сценария установки livecd-iso-to-disk для устройств Live USB, образ корневой файловой системы, rootfs.img , расширяется при установке, и во время загрузки не используется SquashFS.

  • Если в этой ситуации используется rd.live.ram = 1 , полная несжатая корневая файловая система копируется во время загрузки /run/initramfs/rootfs.imgв /runtmpfs.
  • Если rd.live.overlay = none предоставляется в качестве параметра командной строки ядра, при загрузке создается линейный целевой объект Device-mapper с возможностью записи без наложения.
Записываемый образ файловой системы

Система извлечет сжатый образ файловой системы, извлечет его /run/initramfs/fsimg/rootfs.img, подключит к устройству цикла, создаст доступный для записи линейный целевой объект Device-mapper в /dev/mapper/live-rwи подключит его как доступный для записи том в /. Во время загрузки требуется больше ОЗУ, но живой файловой системой легче управлять, если она заполняется. Пользователи могут создавать образ файловой системы любого размера, и этот размер будет сохраняться при загрузке системы. При использовании этой опции изменения корневой файловой системы между загрузками не сохраняются.

Ожидается, что структура файловой системы будет следующей:

rootfs.tgz | Сжатый tarball, содержащий образ файловой системы 
   ! ( 
   Распаковать ) /rootfs.img | Образ файловой системы в / run / initramfs / fsimg / 
      ! ( Монтировать ) 
      / bin | Живая файловая система 
      / загрузка | 
      / dev | 
      ... |

Позволяет использовать модуль ядра OverlayFS , если он доступен, для предоставления объединенного каталога копирования при записи для корневой файловой системы. Наложения OverlayFS - это каталоги файлов, которые были изменены в базовой (нижней) файловой системе, доступной только для чтения. Корневая файловая система обеспечивается с помощью монтирования специального типа наложения, которое объединяет нижний и верхний каталоги. dracut.modules - модули dracut

ОПИСАНИЕ

dracut использует модульную систему для создания и расширения образа initramfs. Все модули находятся в /usr/lib/dracut/modules.d или в <git-src> /modules.d . Самый базовый модуль dracut - 99base . В 99base определен начальный сценарий оболочки init, который запускается ядром после загрузки initramfs. Хотя вы можете заменить init своей версией 99base , это не рекомендуется. Вместо этого следует по возможности использовать крючки дракута. Все хуки и момент времени, в который они выполняются, описаны в разделе «Этапы процесса загрузки» .

Основной скрипт, создающий initramfs, - это сам dracut. Он анализирует все аргументы и устанавливает каталог, в котором все установлено. Затем он выполняет все скрипты проверки, установки и установки ядра, найденные в модулях, которые должны быть обработаны. После того, как все установлено, каталог установки архивируется и сжимается до окончательного образа initramfs. Все вспомогательные функции, используемые check, install и installkernel, находятся в файле dracut-functions . Эти функции оболочки доступны для всех скриптов установщика модулей (install, installkernel), без необходимости исходных dracut-функций .

Модуль может проверить предварительные условия для установки и установки ядра с помощью сценария проверки. Также зависимости могут быть выражены с помощью проверки. Если модуль прошел проверку, будут вызваны install и installkernel для установки всех необходимых файлов для модуля. Чтобы разделить установку на часть ядра и часть, не являющуюся ядром, все части, относящиеся к модулю ядра, должны находиться в installkernel. Все остальные файлы, находящиеся в каталоге модуля, специфичны для модуля и в основном представляют собой сценарии ловушек и правила udev.

Этапы процесса загрузки

Модули dracut могут вставлять собственные скрипты в различные точки для управления процессом загрузки. Эти хуки представляют собой простые каталоги, содержащие сценарии оболочки, заканчивающиеся на ".sh", которые получены с помощью init. Часто используемые функции находятся в файле dracut-lib.sh , который может быть получен любым скриптом.

Крючок: cmdline

CmdLine крюк этого место , чтобы вставить скрипты для разбора командной строки ядра и подготовить более поздние действия, как создание Udev правил и конфигурационные файлы.

В этом хуке определяется самая важная переменная среды: root. Второй - rootok, который указывает, что модуль заявил, что может анализировать определенный корень. Так, например, root = iscsi:…. будет востребован модулем iscsi dracut, который затем устанавливает rootok.

Крючок: pre-udev

Этот хук выполняется сразу после хука cmdline и проверки, установлены ли root и rootok. Здесь модули могут выполнять действия с конечным корневым каталогом и до того, как udev будет запущен.

Запустить Udev

Теперь udev запущен и ведение журнала для udev настроено.

Крючок: предварительный спусковой крючок

В этой ловушке вы можете установить переменные среды udev с помощью udevadm control --property = KEY = value или управлять дальнейшим выполнением udev с помощью udevadm.

Триггер Udev

udev запускается вызовом триггера udevadm, который отправляет события добавления для всех устройств и подсистем.

Основной цикл

В основном цикле dracut зацикливается, пока udev не установится и все скрипты в initqueue / finished не вернут true. В этом цикле есть три перехватчика, куда можно вставить скрипт, вызвав / sbin / initqueue.

Очередь инициализации

Этот хук запускается каждый раз, когда сюда вставляется скрипт, независимо от состояния udev.

Initqueue урегулирован

Этот хук (initqueue / Settlement) запускается каждый раз, когда udev успокаивается.

Время ожидания инициализации

Этот хук (initqueue / timeout) запускается, когда счетчик основного цикла становится половиной счетчика rd.retry.

Initqueue онлайн

Этот хук (initqueue / online) запускается всякий раз, когда сетевой интерфейс подключается к сети (то есть, когда он включен и настроен настроенным сетевым модулем).

Initqueue завершен

Этот перехватчик (initqueue / finished) вызывается после того, как udev установится, и если все скрипты здесь возвращают 0, основной цикл будет завершен. Здесь могут быть добавлены произвольные скрипты для циклического включения очереди инициализации до тех пор, пока что-то не произойдет, чего модуль dracut хочет дождаться.

Крючок: предварительное крепление

Перед монтированием корневого устройства выполняются все сценарии в пре-монтировании хуков. Однако в некоторых случаях (например, NFS) настоящее корневое устройство уже смонтировано.

Крючок: крепление

Этот крючок в основном предназначен для монтирования настоящего корневого устройства.

Крючок: pre-pivot

Этот перехватчик вызывается перед перехватчиком очистки. Это хорошее место для действий, отличных от очистки, которые необходимо вызывать перед поворотом.

Крючок: очистка

Эта ловушка является последней и вызывается до того, как init окончательно переключает root на настоящее корневое устройство. Это хорошее место для очистки и завершения ненужных процессов.

Очистка и switch_root

Init (или systemd) убивает все процессы udev, очищает среду, устанавливает аргументы для реального процесса инициализации и, наконец, вызывает switch_root. switch_root удаляет всю иерархию файловой системы initramfs, chroot () s для реального корневого устройства и вызывает / sbin / init с указанными аргументами.

Чтобы все файлы в иерархии initramfs могли быть удалены, у всех процессов, все еще запущенных из initramfs, не должно быть никаких открытых дескрипторов файлов.

module-setup.sh: check ()

check () вызывается dracut для оценки включения модуля dracut в initramfs.

$ hostonly
Если установлена ​​переменная $ hostonly, то функция модуля check () должна быть в режиме «hostonly», что означает, что check () должен возвращать только 0, если модуль действительно необходим для загрузки этого конкретного хоста.

check () должен вернуться с:

0
Включите модуль dracut в initramfs.
1
Не включайте модуль dracut. Требования не выполнены (отсутствуют инструменты и т. Д.)
255
Включайте модуль dracut только в том случае, если он требуется другому модулю или если он явно указан в файле конфигурации или в списке аргументов.

module-setup.sh: зависит ()

Функция зависит () должна повторять имена всех других модулей dracut, от которых зависит модуль.

module-setup.sh: cmdline ()

Эта функция должна выводить параметры командной строки ядра, необходимые для загрузки текущей настройки машины. Он должен начинаться с пробела и не выводить новую строку.

module-setup.sh: install ()

Функция install () вызывается для установки всего, что не связано с ядром. Для установки двоичных файлов, скриптов и других файлов вы можете использовать функции, упомянутые в [создание] .

Чтобы адресовать файл в текущем каталоге модуля, используйте переменную «$ moddir».

module-setup.sh: installkernel ()

В installkernel () должны быть установлены все файлы, относящиеся к ядру. Вы можете использовать все функции, упомянутые в [создание], для установки файлов.

Функции создания

inst_multiple [-o] <файл> [<файл>…]

устанавливает несколько бинарных файлов и файлов. Если исполняемые файлы указаны без пути, dracut будет искать двоичный файл по пути PATH = / usr / sbin: / sbin: / usr / bin: / bin. Если в качестве первого параметра задана опция «-o», отсутствие файла не приводит к ошибке.

inst <src> [<dst>]

устанавливает один файл <src> либо в то же место в initramfs, либо в необязательный <dst>. inst с более чем двумя аргументами обрабатывается так же, как inst_multiple, все аргументы рассматриваются как файлы для установки, а ни один - как места назначения для установки.

inst_hook <hookdir> <prio> <src>

устанавливает исполняемый файл / скрипт <src> в ловушку dracut <hookdir> с приоритетом <prio>.

inst_rules <udevrule> [<udevrule>…]

устанавливает одно или несколько правил udev. Сообщается о несуществующих правилах udev, но не позволяйте dracut выйти из строя.

instmods <модуль ядра> [<модуль ядра>…]

instmods следует использовать только в функции installkernel ().

instmods устанавливает один или несколько модулей ядра в initramfs. <kernelmodule> также может быть целой подсистемой, если перед ней стоит префикс «=», например «= drivers / net / team».

instmods не установит модуль ядра, если задано значение $ hostonly и модуль ядра в настоящее время не требуется ни для каких / sys / … / uevent MODALIAS. Чтобы установить модуль ядра независимо от режима hostonly, используйте форму:

hostonly = '' instmods <модуль ядра>
